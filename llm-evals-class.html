LLM Evaluations

How to measure whether your language model actually works

10-Slide Crash Course

Why Do Evals Matter?

LLMs are probabilistic — they can look right while being wrong. Without evals, you're flying blind.

🔍 Catch Regressions

A prompt change that fixes one thing can silently break ten others. Evals are your safety net.

📊 Justify Model Choices

GPT-4o vs Gemini vs Claude — evals give you objective data to pick the right model for your task.

⚖️ Regulatory Compliance

EU AI Act, ISO 42001, and NIST AI RMF all require documented evidence of model performance.

🛡️ Safety & Alignment

Evals surface harmful outputs, hallucinations, and bias before users encounter them in production.

💡 Rule of thumb: If you can't measure it, you can't improve it — and you can't defend it.

A Taxonomy of LLM Evals

🤖 Automated vs. 👤 Human

  • Automated — fast, cheap, scalable; string match, embedding similarity, LLM-as-judge
  • Human — gold standard; slow and expensive; best for subjective quality
  • In practice: automate 90%, human-review 10% of edge cases

📌 Functional vs. 🎨 Quality

  • Functional — does it do the task? (tool calling, JSON validity, SQL execution)
  • Quality — how well? (fluency, helpfulness, tone, citation accuracy)
  • Functional evals are binary; quality evals use a rubric or score

📦 Unit vs. 🔗 End-to-End

  • Unit — isolate a single prompt / component / retrieval step
  • E2E — the full pipeline from user query to final response
  • Both are needed — E2E catches integration bugs that unit evals miss

🏆 Benchmark vs. 🧪 Slice

  • Benchmark — standard datasets for comparing models at a high level
  • Slice — subsets of your production data targeting specific behaviors
  • Benchmark scores don't always predict production performance!

Evaluation Methods

🔡 Reference-Based

Compare output against a known-correct answer. Exact match, ROUGE, BLEU, BERTScore. Best for Q&A, summarisation, translation.

🧑‍⚖️ LLM-as-Judge

Use a stronger or same-tier LLM to score outputs on a rubric. Scalable; needs careful prompt design to avoid bias.

🖱️ Human Preference

Side-by-side comparison (A/B) or Likert scale scoring. Gold standard. Powers RLHF and Chatbot Arena leaderboards.

⚙️ Code / Execution

Run the generated code or SQL. Verify it compiles, passes tests, or produces correct query results. HumanEval, LiveCodeBench.

"The right method depends on whether you have ground-truth labels, a budget for human time, and how much you trust a judge model."

Standard Benchmarks You Should Know

🧠 Knowledge & Reasoning

  • MMLU — 57 academic subjects, MCQ
  • BIG-Bench Hard — hard reasoning tasks
  • ARC Challenge — grade-school science
  • GPQA — PhD-level questions

💻 Coding

  • HumanEval — Python function completion
  • MBPP — mostly basic Python problems
  • SWE-bench — real GitHub issues
  • LiveCodeBench — contamination-resistant

🛡️ Safety & Alignment

  • TruthfulQA — detects hallucination
  • ToxiGen — hate speech detection
  • HarmBench — red-teaming attacks
  • IFEval — instruction following
🚨 Benchmark contamination: Models may have seen benchmark data during training. High benchmark scores ≠ strong real-world performance. Always validate on held-out production data.

LLM-as-Judge — Deep Dive

The most practical and scalable approach for production evals.

📥 User Prompt + Model Response (+ Reference Answer if available)
↓
🧑‍⚖️ Judge LLM evaluates against a structured rubric (1–5 scale or pass/fail)
↓
📝 Score + Reasoning
📊 Aggregate Metrics
✅ Best practices: Use chain-of-thought in judge prompt; ask for reasoning before score; use rubrics not vague instructions; calibrate with human labels.
⚠️ Known biases: Position bias (favours first answer), verbosity bias (longer = better), self-enhancement bias (model favours own style). Mitigate with pairwise swap & multi-judge.

  1. Version your prompts — treat prompts like code. Every change should be a commit that re-runs your eval suite.
  2. Separate the eval dataset from training/fine-tuning data — contamination silently inflates your scores.
  3. Gate deployments in CI — block merges if core eval metrics drop below a threshold.
  4. Log production outputs — sample and re-evaluate live traffic to detect distribution shift.

Common Pitfalls & How to Avoid Them

🚨 Overfitting to benchmarks
Teams optimise prompts against the benchmark itself, not the real task. Fix: keep a secret hold-out set and never tune against it.
🚨 Treating all errors equally
A wrong drug dosage and a wrong movie rec are both errors — weigh by business impact. Use severity tiers.
⚠️ Tiny eval datasets
100 examples can't reliably detect a 2% regression. Aim for ≥500 per task slice; use bootstrapping to estimate confidence intervals.
⚠️ Ignoring latency & cost
A 10% accuracy gain that triples cost and latency might not ship. Always report the accuracy-cost-latency frontier, not just accuracy.
💡 Single-metric thinking
Optimising for BLEU alone produces fluent but vacuous text. Always use complementary metrics that pull in different directions.
💡 Skipping failure analysis
A score of 87% tells you nothing. Read 50 failed cases — patterns there drive your next improvement cycle.
