# AI Trust Index — technical reference

> **Status:** Shipped to `develop` (PR included in 2026-06-19 release). Last updated 2026-06-21
> (detail-parity refresh: card-grid UI, feed `indicators`, day-one seed, dropdown counts).

The AI Trust Index is a sixth sidebar module that gives every organisation a read-only window
into an independently-maintained, public feed of AI-app privacy and transparency scores. VerifyWise
pulls the feed on a weekly schedule, detects changes, and emails configured recipients when an
app they track materially changes. No scraping, no LLM, no new external service.

---

## 1. Data source — the feed contract

**Endpoint:** `GET https://verifywise.ai/ai-trust-index.json`
CORS-enabled, served by the verifywise.ai Next.js site (hourly revalidate). Read-only — VerifyWise
never writes back to the website.

### 1.1 Top-level shape

```json
{
  "feedVersion": 2,
  "generatedFrom": "...",
  "meta": { ... },
  "design": { ... },
  "count": 37,
  "apps": [ ... ]
}
```

### 1.2 Per-app shape (all fields present on every row)

```ts
{
  slug: string;               // "claude" — stable identity / join key
  name: string;               // "Claude"
  vendor: string;             // "Anthropic"
  domain: string;             // "claude.ai"
  category: string;           // "Assistant" | "Image & video" | "Audio" | "Companion" | "Productivity"
  scoreOutOf100: number;      // 0–100
  letterGrade: string;        // A | B | C | D | F
  displayedGrade: string;     // letterGrade, capped at B when dealbreaker flags are active
  confidence: string;         // High | Medium | Low
  dealbreakerFlags: string[]; // human-readable violations; often []
  summary: string;            // free-text headline (editorial; prone to rewording)
  highlights: { label: string; text: string }[];  // ~4 findings
  policyUrl: string;          // link to the actual privacy policy
  policyLastUpdated: string | null;  // "YYYY-MM-DD" or null
  modalities: string[];       // ["text"] | ["text","image"] | ...
  processesBiometrics: boolean;
  iconUrl: string;            // derived template — cosmetic only, excluded from hashing
  indicators?: Record<string, {  // optional per-indicator award map (rubric v1.2)
    award: "full" | "half" | "zero";
    subFlag?: "OK" | "SILENT" | "ADVERSE" | "NA";
  }> | null;
}
```

The optional `indicators` map carries the per-indicator award the website assesses each app on
(7 domains, ~30 indicators). It powers the app-detail per-domain breakdown and the
"silent or vague" section. It is additive and optional: apps without it (and pre-`indicators`
DB rows) degrade gracefully. It is not a required field and is not part of either content hash.

### 1.3 Feed versioning

The job accepts **feedVersion 1** and **feedVersion 2**. Version 2 (current live feed) adds a
per-app `history` object (past score snapshots) and the optional `indicators` map above; both are
backward-compatible. Any `feedVersion` outside `{1, 2}` aborts the run — nothing is written to the
database.

| feedVersion | history field | Job behaviour |
|---|---|---|
| 1 | absent | accepted; processed normally |
| 2 | present | accepted; `history` excluded from both hashes |
| ≥3 / unknown | — | abort (write nothing) |

## 2. Architecture

```
BullMQ scheduler (weekly, Mon 06:00 UTC)
  │
  └─► automationWorker dispatch → syncAiTrustIndex action
          │
          ├─ Fetch https://verifywise.ai/ai-trust-index.json
          ├─ Sanity gates (HTTP 200, version check, count integrity, floor check)
          ├─ Canonical hash per app (material + full)
          ├─ Upsert ai_trust_index_apps (one transaction)
          ├─ Soft-delete apps missing from the feed
          ├─ Update ai_trust_index_meta (seeded_at, last_good_count, last_run_week)
          └─ Send digest emails per org (material changes + newly-removed)
```

No Python service is involved. The sync action is a standard Node.js BullMQ worker handler
under `Servers/services/automations/actions/syncAiTrustIndex.ts`.

## 3. Database schema (`verifywise` schema)

Two migrations:

1. `20260619HHMMSS-create-ai-trust-index-tables.js` — creates the four tables below.
2. `20260620125044-seed-ai-trust-index-snapshot.js` — **day-one seed.** On a fresh install this
   loads a committed snapshot of the feed (`Servers/database/seeds/ai-trust-index-snapshot.json`,
   ~140 apps including `indicators`) so the catalog shows data immediately without waiting for the
   first weekly sync. It is idempotent (seeds only when `ai_trust_index_apps` is empty), wraps the
   inserts in one transaction, computes real `material_hash`/`full_hash` via the compiled
   `computeHashes` so the first post-seed sync does not emit a spurious all-apps digest, and sets
   `ai_trust_index_meta.seeded_at`. The snapshot is offline/airgap-safe and is regenerated by
   replacing the JSON file (re-fetch from the live feed).

Both use `queryInterface.sequelize.query` with explicit `verifywise.` prefix.
Models: decorator-based `@Table` classes under `Servers/domain.layer/models/aiTrustIndex/`.

### 3.1 `ai_trust_index_apps` — global feed mirror

**No `organization_id`.** This table is a global cache of the public feed; every organisation
sees the same rows.

### 3.4 `ai_trust_index_meta` — singleton run state

One global row, no primary key beyond a conventional `id`. Never partitioned by org.

| column | type | notes |
|---|---|---|
| `id` | SERIAL PK | always 1 |
| `seeded_at` | TIMESTAMPTZ | set inside the first successful upsert transaction; NULL until then |
| `last_good_count` | INT | count of active apps after the last successful sync |
| `last_run_week` | VARCHAR | ISO week string (e.g. `"2026-W25"`) — singleton guard |

## 4. Change detection — two-hash strategy

### 4.1 Canonical hashing

Hashes are computed over the **incoming parsed feed object**, never re-derived from the stored
`data` JSONB column (Postgres normalises/reorders JSONB keys; re-derivation would flip 100 % of
apps as changed every run).

Before hashing, deep-canonicalise the object:
- Recursively sort object keys.
- Sort semantically-unordered arrays: `modalities` ascending, `dealbreakerFlags` ascending,
  `highlights` by `label` then `text`.
- Use a stable stringify (sorted keys) as the hash input.
- `iconUrl` is always excluded (it is a pure derivative of `domain`; cosmetic only).
- `history` (feedVersion 2 field) is always excluded from both hashes (volatile; changes every
  run and carries no material governance information).

### 4.2 Two hashes — alert-fatigue control

| hash | fields included | effect on email |
|---|---|---|
| `material_hash` | `scoreOutOf100`, `letterGrade`, `displayedGrade`, `dealbreakerFlags`, `policyLastUpdated`, `processesBiometrics` | change → org recipients receive a digest |
| `full_hash` | everything except `iconUrl`, `history`, and the material fields (i.e. editorial copy like `summary`, `highlights`, `confidence`) | change → `data` and `last_changed_at` updated; **no email** |

Reworded summaries or rephrased highlights update the stored record silently. Only a score
change, grade change, new/removed dealbreaker flag, updated policy date, or biometrics toggle
triggers an email.

### 4.3 Per-app upsert (inside one transaction)

```
for each incoming app:
  compute material_hash, full_hash
  if slug not in DB → INSERT (seeded_at guards first-seed email suppression)
  else if full_hash changed → UPDATE data, hashes, last_changed_at
  if material_hash changed (and seeded_at IS NOT NULL) → collect as email candidate

missing slugs (in DB but not in feed, after a sane run):
  UPDATE is_active = false, removed_at = now()     (soft-delete, once)
  if org tracks the slug → collect as "newly removed" candidate
```

## 5. Feed sanity gates (abort-write-nothing)

All gates run before any database write. Failure on any gate aborts the entire run and logs
at `logFailure` level; no rows are modified.

| gate | condition to abort |
|---|---|
| HTTP status | response is not 200 |
| Feed version | `feedVersion` not in `{1, 2}` |
| Count integrity | `feed.count !== feed.apps.length` |
| Floor — absolute | `feed.apps.length < 10` |
| Floor — relative | `feed.apps.length < 0.5 × last_good_count` (skipped on first seed when `last_good_count` is null) |

The relative floor prevents a partial or corrupt feed from silently soft-deleting half the catalog.

## 6. The weekly job

### 6.1 Scheduling

BullMQ job name: `SYNC_AI_TRUST_INDEX` (or equivalent constant in `automationProducer.ts`).
Cron expression: `0 6 * * 1` (Monday 06:00 UTC).
Registered last in `Servers/jobs/producer.ts` to avoid blocking other schedulers at startup.

### 6.2 First-seed guard (`seeded_at`)

- `seeded_at` in `ai_trust_index_meta` starts NULL.
- The job sets `seeded_at = now()` **inside** the first successful upsert transaction.
- Email candidates are collected only when `seeded_at IS NOT NULL` (checked *before* the
  transaction commits). This ensures the initial bulk insert never triggers digest emails.

### 6.3 Singleton guard (`last_run_week`)

Before fetching the feed, the job reads `last_run_week` from `ai_trust_index_meta`. If it
matches the current ISO week, the job exits immediately (no fetch, no write). This prevents
double-runs from BullMQ job restarts, worker crashes, or manual re-triggers within the same
calendar week.

### 6.4 Handler flow (summary)

```
1. Read last_run_week → exit if same week
2. Write last_run_week = current week (optimistic; prevents parallel double-run)
3. Fetch feed → sanity gates
4. For each app: compute hashes, collect material-changed + newly-removed
5. Upsert all apps in one transaction; set seeded_at on first run
6. Soft-delete missing apps
7. Update last_good_count
8. If seeded_at was already set: resolveRecipients per org → send digests
```

## 7. Email digest

### 7.1 Recipient resolution per org

```
configured_users  = users in recipient_user_ids who still belong to the org (re-validated at send time)
configured_emails = recipient_emails (format-validated at settings save; re-used as-is)
all_recipients    = configured_users.emails ∪ configured_emails

if all_recipients is empty:
  → fall back to all org users whose role is Admin (role_id = 1) or SuperAdmin
```

Cross-org isolation: each org's recipient list is resolved independently; a material change in
org A's tracked app never emails org B.

### 7.2 Digest content

Two sections in a single email per org per run. Each line shows the human app **name** (not the
slug), resolved via a join in `getAffectedOrgsBySlugs`:

- **Changed apps** — for each tracked app whose `material_hash` changed, the name plus its current
  grade, e.g. `Paxton AI — now grade B`.
- **No longer assessed** — for each tracked app newly missing from the feed (`removed_at` just
  set this run), the name only. Guard: if `removed_at` was already set in a prior run, the app is
  NOT re-collected (no repeat email for a persistently-removed app).

If the join finds no apps row (rare), the slug is used as a fallback name. Names are HTML-escaped
before MJML interpolation.

Template: `Servers/templates/ai-trust-index-digest.mjml` (rendered via the existing MJML
pipeline, sent via `sendAutomationEmail`).

## 8. REST API

Base path: `/api/ai-trust-index`
Auth: all routes behind `authenticateJWT`.
Pattern: thin controller (`aiTrustIndex.ctrl.ts`) → utils (`aiTrustIndex.utils.ts`).
Response envelope: `STATUS_CODE[200](data)`.
Logging: `logProcessing / logSuccess / logFailure`.

| method + path | purpose | notes |
|---|---|---|
| `GET /apps` | paginated browse | query params: `search, category, grade, page, pageSize, sort(score\|name\|vendor\|category), dir(asc\|desc)`. `pageSize` is bounded at 1000 (so a consumer can request the whole catalog in one page — the app-detail comparison strip does this for an accurate rank and category average). Server-side LIMIT/OFFSET + parallel COUNT. Each row includes `isTracked` for the caller's org via LEFT JOIN tracked. Returns `{ apps, total, page, pageSize, categories, categoryCounts, gradeCounts }`. `categoryCounts`/`gradeCounts` are computed over the whole active catalog (not the current filter) so the Browse dropdowns can show stable per-option counts. Only `is_active = true` rows. |
| `GET /apps/:slug` | app detail | full `data` JSONB + `isTracked`. If slug is inactive but the org tracks it → return with `noLongerInIndex: true` (never 404 a tracked-but-removed app). |
| `GET /tracked` | org's tracked apps | LEFT JOIN tracked → apps so removed-but-tracked rows appear with `noLongerInIndex` flag; never INNER-JOIN them away. |
| `POST /tracked` | track one app | body `{ slug }`. Validates slug exists and `is_active`. Idempotent via UNIQUE(org, slug). |
| `POST /tracked/bulk` | track many | body `{ slugs: string[] }`. Normalise + validate each; insert all in one transaction; skip already-tracked or inactive slugs. Returns `{ tracked: string[], skipped: string[] }`. List capped at ≤ 200 slugs. |
| `DELETE /tracked/:slug` | untrack | scoped to caller's org. |
| `GET /settings` | get recipients | returns `{ recipientUserIds, recipientEmails }`. |
| `PUT /settings` | update recipients | **Admin / SuperAdmin only** (role guard). Validates email format + that `recipientUserIds` belong to the org. Upserts the settings row. |

## 11. Testing

- **Canonical hashing:** same object with reordered keys/arrays → same hash (regression guard).
- **Two-hash semantics:** reworded `summary` → `full_hash` changes, `material_hash` unchanged → no email.
- **Sanity gates:** empty feed / count mismatch / unknown feedVersion / below-floor count → abort, no writes.
- **First-seed guard:** empty meta → seeds + sets `seeded_at`; no emails sent.
- **Soft delete / reappearance:** missing app → `is_active = false`, `removed_at` set; reappears → `is_active = true`, `removed_at` cleared.
- **Removed-app email once-only:** app removed → in `newlyRemoved` this run; next run while still gone → not re-collected.
- **Bulk track:** `POST /tracked/bulk` inserts new slugs, skips already-tracked + inactive; over-cap rejected.
- **Recipient resolution:** users ∪ free-text; empty → Admin fallback; cross-org isolation.
- **Pagination/search util:** filter + LIMIT/OFFSET + total count.
- **Frontend (Vitest):** Browse renders + filters call API with params; Track toggle optimistic + invalidation; Settings Admin-gating; AppDetail renders `displayedGrade` and `noLongerInIndex`.
