# AI Trust Index — App Detail Parity (design)

> **Date:** 2026-06-20
> **Status:** Approved design — pending implementation plan
> **Branch (intended):** `feat/ai-trust-index` (continuation) or a fresh `feat/ai-trust-index-detail-parity`

## Goal

Bring the AIPurview **app** AI Trust Index detail page (`Clients/.../pages/AITrustIndex/AppDetail/index.tsx`)
up to visual + informational parity with the **public website** detail page
(`website/verifywise/app/ai-trust-index/[slug]/page.tsx`).

The website detail page is rich and lively; the app detail page is a lean card stack. We want the app
page to show the same set of sections, driven by the same data, rendered with AIPurview components and
design tokens.

## Background — the two implementations

| | Public website | AIPurview app |
|---|---|---|
| Stack | Next.js (static) | React 19 + Vite + MUI |
| Data source | `lib/ai-trust-index-data.ts` + `ai-safety-index/history/*.json` | DB (`ai_trust_index_apps.data` jsonb), synced weekly from the hosted feed |
| Detail page | `app/ai-trust-index/[slug]/page.tsx` | `pages/AITrustIndex/AppDetail/index.tsx` |

Both consume the same per-app record shape (slug, name, vendor, domain, category, score, grades,
confidence, dealbreakerFlags, summary, highlights, policy meta, modalities, processesBiometrics).

### The data chain

```
website repo                          published feed                 AIPurview app
ai-safety-index/history/*.json   →   verifywise.ai/                →  ai_trust_index_apps.data (jsonb)
  (30-indicator award map)            ai-trust-index.json              →  AppDetail renders
                                       (feedVersion 2)
```

- The app's feed URL is **hardcoded**: `FEED_URL = "https://verifywise.ai/ai-trust-index.json"`
  (`Servers/utils/aiTrustIndexFeed.ts:5`). Every install — cloud or self-hosted — pulls the same upstream.
- The feed is generated by the website route `app/ai-trust-index.json/route.ts`, which already reads the
  history files and emits `feedVersion: 2` with a per-app `history` object — but **does not emit the
  per-indicator award map** that drives the website's two richest sections. (App count is data, not
  architecture — ~99 apps as of 2026-06-20; nothing in this design hardcodes a count.)
- The app persists the feed into Postgres via a **weekly** BullMQ cron (Mon 06:00 UTC,
  `automationProducer.ts:221`, scheduled at boot from `jobs/producer.ts:31`). It self-guards via
  `last_run_week`. The app reads from its own DB (`getAppBySlugQuery` → `SELECT a.*`), so the stored
  `data` jsonb flows verbatim to the UI. React Query caches it only briefly (`staleTime: 2s`,
  `gcTime: 10min`).

## Section gap (website has, app lacks)

| Section | Website | App today | Buildable in app from… |
|---|---|---|---|
| Grade hero + **score meter bar** | ✅ | ❌ (chip + text only) | existing data |
| Verdict line ("earns a B because…") | ✅ | ❌ | existing data |
| Comparison strip (rank #N, vs category avg) | ✅ | ❌ | existing data (needs full app list) |
| Grade-scale A–F chips | ✅ | ❌ | design tokens (grade bands) |
| **Watch-outs** (silent/vague) | ✅ | ❌ | **indicator map (NOT in feed yet)** |
| **Score breakdown** (per-domain bars + 30-indicator checklist) | ✅ | ❌ | **indicator map (NOT in feed yet)** |
| Related apps (same category) | ✅ | ❌ | existing data |
| Track / Untrack toggle | ❌ (public) | ✅ | app-only, keep |

## Scope decision

**Full parity** — including the indicator-driven sections. This requires threading the indicator award
map from the website history files, through the published feed, into the app DB, and rendering it.

## Architecture — the split

### 1. Website repo (`website/verifywise`) — user commits

**One file:** `app/ai-trust-index.json/route.ts`.
- Add an `indicators` field to each emitted app object, read from the *same* history file the route
  already opens for `history` (`assessments[last].indicators`), mirroring the detail page's `readIndicators`.
- Shape: `Record<string, { award: "full" | "half" | "zero"; subFlag?: "OK" | "SILENT" | "ADVERSE" | "NA" }>`.
- Keep **`feedVersion: 2`** — `indicators` is additive/optional; existing consumers ignore unknown fields.
- Apps without a history file emit `indicators: null` (same as `history`).

This is the entire upstream change. (Out of this repo's control; documented here for the chain to make sense.)

### 2. App backend (`Servers`) — this repo

**a. Interface** — add one optional field to `ITrustIndexAppData`
(`Servers/domain.layer/interfaces/i.aiTrustIndex.ts:4`):
```ts
indicators?: Record<string, { award: "full" | "half" | "zero"; subFlag?: "OK" | "SILENT" | "ADVERSE" | "NA" }> | null;
```
- `validateFeed` filters apps by REQUIRED_KEYS only and passes the **whole object** through; `upsertFeedTx`
  stores `JSON.stringify(app)` into `data` jsonb. So the data flows with **no query, validation, or
  migration change**. The optional field just makes it type-safe instead of `as any`.
- `indicators` is **not** a required key and **not** part of either content hash (it's derived from the
  same assessment, so it co-varies with material fields; adding it to the hash is unnecessary and would
  churn change-detection on the first synced feed). Leave `computeHashes` untouched.

**b. Day-one seed snapshot** (new migration) — so a fresh install shows data immediately, offline/airgapped,
without waiting for the first Monday cron.
- Commit a snapshot JSON of the feed (the full current app set — ~99 apps as of 2026-06-20, including
  `indicators`) under `Servers/database/seeds/ai-trust-index-snapshot.json`. The migration reads this file
  (not inlined) so regeneration is a file replace, not a migration edit.
- New migration (timestamped) inserts the snapshot rows into `ai_trust_index_apps` **only if the table is
  empty** (guard on `seeded_at IS NULL` / `NOT EXISTS (SELECT 1 FROM ai_trust_index_apps)`), then sets
  `ai_trust_index_meta.seeded_at` so the weekly sync treats the next run as a normal (non-first) run.
- Follows the existing seed-migration pattern (`20260302111132-seed-framework-struct-data.js` et al.),
  uses `verifywise.` DDL prefix, idempotent (`ON CONFLICT (slug) DO NOTHING`).
- The weekly cron refreshes it thereafter; the snapshot is the floor, not the ceiling.
- **Snapshot staleness is accepted** (per decision): it ages in the repo and is regenerated periodically.
  A short README note documents how to regenerate it from the live feed.

> No manual "sync now" endpoint. Staged rollout is acceptable: website feed change → next weekly sync
> picks it up. The seed covers the install-time "show something now" requirement.

### 3. App frontend (`Clients`) — this repo

**a. Rubric module** — `Clients/src/presentation/pages/AITrustIndex/rubric.ts`.
Port from the website's `lib/ai-trust-index-rubric.ts` (source of truth):
- `RUBRIC_DOMAINS`, `INDICATOR_LABELS` (30), `INDICATOR_GAP_LABELS` (30), `AWARD_LABELS`, `SUBFLAG_LABELS`
- types `Award`, `SubFlag`, `IndicatorAward`, `IndicatorMap`
- `summarizeDomains(indicators)` → per-domain `{ full, half, zero, na, applicable, ratio }`
- **Note:** carry over the website's *fixed* labels (the `half` → "Partial" fix from commit `36622e94`,
  and the gap-label set), so the app does not reintroduce the bugs already fixed upstream.

**b. Types** — extend `TrustIndexAppData` in `pages/AITrustIndex/shared.ts` with optional
`indicators?: IndicatorMap | null` (and surface `confidence`, already present in the type but unused).

**c. Rebuild `AppDetail/index.tsx`** to the website section order, mapped to AIPurview primitives:

1. Breadcrumbs + "← Back to Browse" (keep existing)
2. **Grade hero** — large letter grade on its grade-variant color, favicon + name + vendor, **score meter
   bar**, band label + confidence. Track/Untrack button stays in the header (app-only).
3. **Verdict line** — grade-aware sentence (port `VERDICT_QUALITY`).
4. Capped-grade note (when `displayedGrade !== letterGrade`).
5. Dealbreaker flags (keep; styled as the website's alert or the existing error chips — pick the alert look).
6. **Comparison strip** — `#rank of N`, score vs category average **excluding self** (carry the
   `36622e94` peer-average fix), `>= 2` peers to show.
7. **Grade scale** — A–F band chips, active one highlighted.
8. Summary.
9. **Highlights** — 2-col cards (keep).
10. **Watch-outs** — "what the policy is silent or vague on", from `indicators`; render nothing when no
    indicators or no gaps.
11. **Score breakdown** — per-domain bars + 30-indicator checklist with dot legend; fallback note when
    `indicators` is null/empty (mirror the website's "being prepared" copy).
12. Policy details (keep) → **Related apps** (same-category internal links with grade badges).
13. Scope/disclaimer line.

**d. Sub-components** under `pages/AITrustIndex/`:
- `AppDetail/ScoreBreakdown.tsx` (port of website `score-breakdown.tsx`, VW-styled)
- `AppDetail/insights.tsx` — `VerdictLine`, `WatchOuts`, `ComparisonStrip`, `RelatedApps`
  (ports of website `app-insights.tsx`, VW-styled)
- Each takes plain props (the app record / indicator map); no app-specific data fetching inside.

**Design-token mapping (website hex → VW tokens):**
- Grade colors → existing `gradeVariant` Chip variants + `GRADE_STYLES` from the feed `design` block (the
  feed already ships `design.gradeStyles` / `design.gradeBands`; prefer those over hardcoding).
- Borders `#d0d5dd` → `palette.border.dark`; text → `palette.text.tertiary` / `theme.palette.text.secondary`.
- Brand green `#13715B` → `palette.brand.primary`.
- Amber watch-out / bar colors → reuse the website ramp but pull through tokens where an equivalent exists.

## Graceful degradation

- App record **without** `indicators` (pre-sync DB rows, or apps the website hasn't graded on the indicator
  rubric): Watch-outs + Score breakdown render the fallback note; everything else (hero, verdict,
  comparison, grade scale, highlights, related) still renders from base data.
- This is the same fallback the website uses, so the two stay visually consistent during the rollout window.

## Data flow (after rollout)

```
website feed (indicators added)
   → weekly Mon 06:00 cron OR install-time seed snapshot
   → ai_trust_index_apps.data jsonb (indicators inside)
   → getAppBySlugQuery SELECT a.*  (verbatim)
   → useApp(slug) (React Query, 2s stale)
   → AppDetail renders all sections; indicator-driven sections degrade if absent
```

## Testing

- **Frontend (Vitest):**
  - `summarizeDomains` math (full=1, half=0.5, zero=0 over applicable; NA excluded; ratio).
  - `VerdictLine` renders a grade-consistent sentence for each grade.
  - `WatchOuts` returns null with no indicators / no gaps; lists gap phrases otherwise (uses GAP labels).
  - `ComparisonStrip` excludes self from peer average; shows "—" with < 2 peers.
  - `ScoreBreakdown` renders fallback when `indicators` null; renders bars + checklist when present.
  - `AppDetail` renders all base sections without `indicators` (degradation path).
- **Backend (Jest):**
  - `validateFeed` preserves `indicators` on the returned app object.
  - seed migration is idempotent and only seeds when the table is empty (logic-level test or documented manual check).
- **i18n:** any new user-facing strings (section titles, fallback copy, legend labels) added to
  `de/fr/es` so `i18n:audit:strict` passes.

## Out of scope

- Manual "sync now" endpoint (staged rollout accepted).
- Changing the weekly cadence or the hash algorithm.
- Website-side work beyond the one feed-route field (the user commits the website repo).
- Score-over-time / history trend UI (feed carries `history`, but trend visualization is a separate feature).

## Risks / notes

- **Snapshot freshness:** the bundled seed ages; document regeneration. Not a correctness risk (cron
  refreshes), only a "first impression on a brand-new install before first cron" freshness gap.
- **Indicator absence window:** until the website feed ships `indicators` AND a sync runs, live DB rows
  won't have indicators; the seed snapshot can include them from day one, so seeded installs show the rich
  sections immediately while non-seeded/older rows degrade. Acceptable and self-healing.
- **Label drift between repos:** the app's `rubric.ts` is a *port*. If the website rubric changes, the app
  copy can drift. Mitigation: a comment pointing to the website source of truth + the labels also ship in
  the feed `design` block where practical (future: move all rubric labels into the feed to remove the port).
